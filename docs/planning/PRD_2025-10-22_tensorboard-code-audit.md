# PRD — TensorBoard 전수 코드 검수

## Executive Summary
TensorBoard 기능 전반(백엔드 API, 모델 내보내기 스크립트, 프런트엔드 UI, 배포 스크립트)을 재검수하여 더미 데이터 링크, 중복 호출, 누락된 검증 로직 등을 제거한다. 목표는 사용자가 최신 모델 임베딩과 지표를 안정적으로 확인할 수 있도록 전체 라인 흐름을 안전하게 정리하는 것이다.

## Problem Statement
현재 TensorBoard 관련 코드가 여러 위치에 흩어져 있고, 일부는 더미 모델/링크를 유지한 채로 남아 있다. 또한 중복 API 호출이나 중복 링크가 존재할 가능성이 있으며, 잘못된 경로 지정으로 인해 실사용자가 더미 데이터를 받는 문제가 재발할 수 있다. 전 repo를 대상으로 전수 점검하여 정확한 현황과 개선책을 제시해야 한다.

## Goals and Objectives
- TensorBoard 관련 코드/설정/문서를 100% 식별하여 소유 경로를 분류한다.
- 중복 호출, 중복 링크, 더미 자산 경로 여부를 각각 최소 1회 이상 검증한다.
- 식별된 문제를 정량 지표(예: 중복 호출 수, 더미 링크 수, 고쳐야 할 파일 수)로 보고하고 해결 우선순위를 제시한다.

## Requirements
1. 백엔드 API 및 서비스 계층에서 TensorBoard를 언급하는 함수 호출과 라우트를 모두 나열하고 의존 경로를 문서화한다.
2. 프런트엔드 여러 패키지(home/prediction/training/shared 등)에서 TensorBoard 관련 컴포넌트와 링크를 추적하여 실제 백엔드 호출 흐름과 일치 여부를 확인한다.
3. 모델 내보내기 스크립트와 훈련 파이프라인 전반에서 더미 데이터 활용 여부, 검증 로직 부족 등을 체크한다.
4. 템플릿/문서/배포 스크립트에서 TensorBoard 자산 경로가 최신 경로를 반영하는지 검증한다.
5. 중복 호출/링크/더미 데이터 현황을 CSV 표 수준으로 요약하고 각 항목당 개선 제안 및 예상 효과를 계산한다.

## Phase Breakdown
- **Phase 1 — 인벤토리 수집**: 전 repo에서 TensorBoard 키워드 검색, 파일별 역할 분류, 목록화.
- **Phase 2 — 문제 분석**: 중복 호출, 더미 경로, 비정상 링크 여부 검토 및 로그/코드 교차 확인.
- **Phase 3 — 개선안 도출**: 정량 지표 산출, 영향도 평가, 권장 변경 사항 및 테스트 전략 마련.

## Success Criteria
- TensorBoard 관련 코드 파일 100%가 인벤토리 표에 등재되고 분류됨.
- 중복 호출·링크·더미 자산 여부가 각 1건 이상 검증되어 상태가 보고됨.
- 개선안이 정량 지표(예: 제거 대상 더미 링크 N개, 통합 가능한 API 호출 M건, 로그 검증 기준 등)와 함께 제시됨.

## Timeline Estimates
- Phase 1: 3.0 h — 코드 스캔 및 분류, 인벤토리 작성.
- Phase 2: 3.5 h — 라우트/컴포넌트 분석, 중복 및 더미 확인.
- Phase 3: 2.5 h — 요약 보고, 개선안/KPI 설계.
- 총 예상: 9.0 h (2025-10-23 중 보고 목표).
